## What was broken

The bali production proxy (`outpost-testnet-miden-testnet/miden-agglayer-0`,
running `gatewayfm/miden-agglayer:0.2.1`) had been rejecting every
aggoracle `insertGlobalExitRoot` push for ~20 wall-days with:

```
src/service.rs:396 ERR eth_sendRawTransaction: account data wasn't found
for account id 0xe9a21e616d9ed59016d481c7001393
```

2 stuck L1→bali deposits — `cnt=1130654` (marti) and `cnt=1131034`.

## What we found (timeline)

Verified from Loki + code archaeology + e2e repro:

| When | Event |
|------|-------|
| **2026-04-13** | Commit `dbe5c2d` merged `add_public_wallet` into `add_wallet` during the 0.14.x migration, accidentally dropping `.storage_mode(AccountStorageMode::Network)`. All accounts deployed from this point on default to `Private` (no on-chain copy). |
| **2026-04-23** | bali deployed via the buggy code. ger_manager, bridge, service all Private. |
| **2026-05-11 16:14:51Z** | First `incorrect account initial commitment` (IAIC) log line on bali. Triggered by concurrent claim+GER submissions racing the `bridge` account's mempool slot (the proxy maintained two miden-client instances against the same sqlite — long-lived for GER, fresh-per-claim for claims). Recurred 189 times through 2026-05-14. |
| **2026-05-14 18:45:18Z** | Operator ran `--reset-miden-store --restore` to recover. `--reset` wiped `store.sqlite3`; `--restore` Phase 1 only called `sync_state()` — does NOT re-import bridge accounts. `latest_account_headers` came back empty. |
| **2026-05-14 18:45:18Z onwards** | Every aggoracle push now fails with `AccountDataNotFound`. Backlog grows to 2 deposits including marti's. |

## What v0.3.0 ships (13 commits + 1 merge over `main`)

### Cure for the active failure mode

1. **`fix(init): use AccountStorageMode::Public for add_wallet`** — restores the dropped storage_mode call. Future deployments are recoverable.
2. **`feat(self-heal): runtime retry on AccountDataNotFound + IAIC`** — typed-downcast catch on `ClientError` + `import_account_by_id` from node + retry once. Catches both error types via the same code path. 5 unit tests pin the typed match against canonical upstream variants.
3. **`feat(restore): import bridge accounts in Phase 0 of --restore`** — fixes the regression chain. `--reset-miden-store --restore` now actually re-imports bridge accounts before `sync_state()`. The repro `e2e-reset-restore-recovery.sh` proves both the bug (pre-fix) and the cure (post-fix) end-to-end.

### Structural prevention of the root-cause race

4. **`feat(claim): route publish_claim through long-lived MidenClient`** — eliminates the dual-sqlite-handle race. Every Miden submission now funnels through the long-lived client's `mpsc::channel(1)`. At most one in-flight submission per node ⇒ structurally impossible to produce two mempool-conflicting txs from the proxy.
5. **`feat(indexer): persistent cursor + resume on restart`** — closes the "indexer strands GERs across pod restart" gap that produced bali's 27 historic STATE-C orphans.

### Defensive hardening + operational fixes

6. **`fix(logging): clamp alloy http transport to info`** — stops `RUST_LOG=debug` from logging the Sepolia archival `?apiKey=` in every L1 RPC line. Active SIEM credential leak.
7. **`fix(accounts-config): atomic write for bridge_accounts.toml`** — fsync + rename. Prevents OOM-mid-write corruption.
8. **`feat(startup): in-process DB migrator`** — embeds migrations via `include_str!` + checksum-verified `schema_migrations` + advisory lock. Replaces the `agglayer-migrate` compose service that was a deploy-time footgun (new migrations had to be wired into compose by hand).
9. **`feat(indexer): --l1-indexer-from-block override`** — operator backfill flag for the 27 historic STATE-C orphans on bali (cosmetic; non-blocking).

### Tests + docs

10. **`docs(runbook): bali v0.3.0 deploy + recovery procedure`** — SRE operational guide. Explicit "bali existing accounts are Private; full redeploy is the cure for the current pod" note.
11. **`test(e2e): self-heal repro + main evidence`** — `scripts/e2e-account-self-heal.sh` with `MODE=expect_failure / expect_self_heal`. Same script proves bug on `main` AND cure on this branch.
12. **`test(e2e): operator-faithful reset+restore + IAIC mempool repros`** — two more repros for the additional failure modes.
13. **`test(e2e): refresh evidence files`** — committed proxy log captures.

## Security review

Independent sub-agent audited each commit + Cantina regressions + auth paths. Verdict: **SHIP**.

- `service.rs`, `service_send_raw_txn.rs`, `service_admin.rs`, `service_state.rs` — byte-identical to main on the auth paths. R1 admin auth, R2 allowed_signers, R13 rate limit untouched.
- Cantina #1–#13 all PASS (untouched paths).
- `AccountStorageMode::Public` flip on `add_wallet`: SAFE. BasicWallet account storage holds no secrets — private keys live in `FilesystemKeyStore` regardless of storage_mode.
- Net security delta POSITIVE: closes the active `RUST_LOG=debug` apiKey leak.

## Bali deployment plan

See `docs/REDEPLOY_RUNBOOK_BALI.md`. Key point:

**v0.3.0's runtime self-heal CANNOT recover bali's current ger_manager** because that account is Private (created pre-`dbe5c2d`-fix). `import_account_by_id` returns `AccountIsPrivate`. Bali specifically needs a **full redeploy** with fresh Public accounts + aggkit reconfig — runbook documents the procedure.

For all FUTURE deployments (and any other proxy that re-deploys with v0.3.0 since this PR), the bug class is closed:
- Future accounts are Public → recoverable from a sqlite loss.
- Runtime self-heal catches the recovery automatically.
- Dual-cache race is structurally impossible.
- `--reset-miden-store --restore` works end-to-end.
